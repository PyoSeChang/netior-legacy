# 새출발

## Problem

기존 Netior의 도메인 모델이 너무 복잡해졌다.

- `meaning`, `field`, instance 저장 모델, network engine이 서로 얽히면서 무엇을 해결하려는 모델인지 흐려졌다.
- 초기에는 이 구조로 실제 ontology를 구성할 수 있다고 믿었지만, 설정 비용이 크고 실제 동작 감각이 약했다.
- 겉으로는 그럴듯하지만, 사용자가 자기 세계를 정의하고 변화시키는 데 충분히 단순하고 강력한 언어가 되지 못했다.

가장 큰 문제는 Netior가 instance와 그 내용을 너무 많이 소유하려 했다는 점이다.

- `.md`, `.svc`, 이미지, 코드 파일, JSON 데이터, 외부 서비스 객체를 모두 Netior instance로 만들려고 하면 부담이 커진다.
- 사용자의 세계는 이미 파일, 폴더, 서비스, 문서, 이미지, 코드 안에 흩어져 있다.
- Netior가 모든 내용을 자기 DB 안으로 끌어들이면, 외부 세계와의 연결성이 약해지고 ontology가 오히려 부정확해진다.
- Netior가 해야 할 일은 resource를 소유하는 것이 아니라, 그 resource가 어떤 종류인지, 어떤 instance의 근거인지, 어떤 관계에 참여하는지를 기록하는 것이다.

또 하나의 문제는 닫힌 구조가 `meaning`과 field behavior를 과도하게 복잡하게 만들었다는 점이다.

- 기존 구조에서는 Netior가 도메인 독립적으로 모든 해석, 계산, 검증을 처리해야 한다는 압력이 있었다.
- 그 결과 `schema_field_bindings`는 option source, schema composition/extension, conditional field, computed field, derived collection까지 떠안는 층이 되었다.
- 이 방향은 표현력은 있지만, 사용자가 설정하기 어렵고 Netior 내부 모델을 무겁게 만든다.
- 닫힌 Netior가 모든 도메인 규칙을 일반화하려는 순간, meaning은 의미의 언어가 아니라 범용 계산/행동 엔진의 대체물이 된다.

방치하면 Netior는 복잡한 ontology editor도, 실용적인 파일 탐색기도, AI-native world tool도 아닌 애매한 앱이 된다.

## Context

이번 세션에서 다음 전제가 새로 정리되었다.

- 사용자는 World의 최종 권위자다.
- AI는 사용자의 세계관을 알 수 없으며, 비대칭 정보가 있다.
- Netior는 사용자가 정의한 World를 기술하고, instance/resource에 적용하고, 변화와 판단을 기록하고, 보여주는 앱이다.
- AI는 translator, actor, validator 같은 역할을 가질 수 있지만, 이 역할들은 서로 격리되어야 한다.
- Netior는 instance content의 소유권을 크게 내려놓되, schema, relation, assignment, evidence, view, measurement의 책임은 더 명확히 가져야 한다.

기존 네트워크 철학은 버리지 않지만, network engine이 모든 것을 해결해야 한다는 기대는 낮춘다.

- 사람은 개념을 tree가 아니라 관계망으로 기억한다.
- 하지만 Netior가 instance contents를 소유하지 않는다면, world는 directory path와 더 직접적으로 연결되어야 한다.
- World/Model은 tree 구조를 가질 수 있어야 하고, directory 하위 구조와의 관계도 명시되어야 한다.
- network는 철학이자 projection 방식이다. 저장과 탐색의 기반 전체를 canvas engine 하나에 맡기지 않는다.

legacy 확인 결과, 기존 field binding은 실제로 닫힌 구조의 압력을 보여준다.

- `field_type`은 text, number, select, object, file 같은 저장/편집 primitive를 나타냈다.
- `schema_field_bindings`는 그 필드가 어디서 해석되거나 채워지는지를 나타냈다.
- 같은 binding 계층이 `instance_select`, `schema_composition`, `schema_extension`, `conditional_field`, `computed_field`, `derived_collection`을 함께 담았다.
- 이는 Netior 내부에서 도메인 규칙과 계산까지 흡수하려는 설계였고, 새 방향에서는 이 부담을 줄여야 한다.

## Decision

Netior를 새 방향으로 재정의한다.

- Netior는 instance DB가 아니다.
- Netior는 정적인 ontology editor도 아니다.
- Netior는 닫힌 저장소가 아니라, 외부 resource 위에 ontology를 부여하고 변화의 근거를 축적하는 열린 world tool이다.
- Netior는 사용자가 World 안의 종류와 관계를 정의하고, 그것을 instance/resource에 할당하고, 그 결과와 변화를 보여주고, AI와 함께 해석하는 앱이다.

결정을 한 문장으로 요약하면:

```text
Netior는 World 안에 무엇이 존재하고, 그것들이 어떤 관계를 맺는지 정의한 뒤, 그 정의를 외부 resource와 instance에 할당하고, 그 변화의 근거를 축적해 보여주는 AI-native world tool이다.
```

이번 세션에서 잡은 큰 방향:

- 앱 이름은 Netior로 유지한다.
- `World`는 유지하고, 하위 개념은 `Model`로 부른다.
- `Scope`, `Realm`은 쓰지 않는다.
- `Kind`, `Property`, `RelationKind`, `Instance`, `Resource`를 핵심 언어로 삼는다.
- `Property`는 value를 가지지 않는다. Value는 Instance에 속한다.
- Instance와 Resource는 분리한다.
- Instance는 ontology target이고, Resource는 파일, 폴더, URL, service object, sub-resource 같은 외부 근거다.
- Netior는 resource content를 직접 소유하지 않는다. 대신 resource와 instance의 매핑, kind assignment, relation assignment, change evidence를 가진다.
- Meaning은 가능한 한 단순해져야 한다. 도메인별 결정 규칙과 계산은 Netior 내부 meaning 모델이 아니라 capability/SDK/외부 실행 계층으로 열어둔다.
- View는 이 정의와 할당, 변화 근거를 보여주는 projection이다. View 자체가 핵심 변화는 아니며, 이 로그의 중심도 아니다.

## Rationale

Netior의 핵심은 사용자가 자기 World를 기술하고 움직일 수 있게 하는 것이다.

사용자는 자신이 쓰려는 소설의 세계관, 등장인물, 플롯, 연구 자료, 프로젝트 규칙을 가장 잘 안다. AI가 먼저 아는 것이 아니다. 따라서 Netior의 언어는 AI의 똑똑함에 기대기보다, 사용자가 세계를 명시할 수 있는 구조여야 한다.

Instance 소유권을 내려놓는 것은 기능 축소가 아니라 개방성의 회복이다.

- 파일 내용은 파일에 있다.
- 이미지 내용은 이미지에 있다.
- 코드와 설정 값은 코드 파일과 설정 파일에 있다.
- 데이터는 CSV, JSON, DB, API에 있을 수 있다.
- HTML interactive view는 HTML resource에 있다.
- 서비스 객체는 외부 서비스에 있다.
- Netior는 그것들을 강제로 자기 instance로 복제하지 않고, World의 언어로 해석하고 연결하고 검증한다.

이렇게 열어둘 때 ontology는 더 풍부하고 정확해질 수 있다.

- 모든 resource를 Netior-owned instance로 만들지 않아도 된다.
- `.md`, `.svc`, 코드 파일, 이미지, 데이터 파일은 원래 위치와 형식을 유지한다.
- Netior는 그 resource가 어떤 Kind로 읽히는지, 어떤 Instance의 근거인지, 어떤 Relation에 참여하는지만 기록한다.
- 따라서 외부 도구와의 호환성을 잃지 않으면서도, 사용자의 world model은 더 정밀해진다.

Meaning 간소화도 같은 이유에서 가능해진다.

- 닫힌 Netior에서는 도메인 독립성을 내부 모델 하나로 해결하려 했기 때문에 meaning과 field binding이 복잡해졌다.
- 하지만 열린 Netior에서는 도메인별 결정 규칙, 계산, 검증을 외부 capability로 보낼 수 있다.
- 예를 들어 "a 필드는 b 필드보다 값이 1 작아야 한다"는 규칙은 Netior의 보편 지식이 아니라 해당 domain의 규칙이다.
- Netior는 그 규칙을 직접 일반화하기보다, 규칙을 어디에 맡기고 결과를 어떻게 기록할지를 정의해야 한다.

## Alternatives Considered

### Instance를 계속 강하게 소유한다

- 내용: Instance를 별도 DB 모델로 강하게 저장하고, Netior가 instance content와 lifecycle까지 주도한다.
- 장점: 내부 일관성과 query는 단순할 수 있다.
- 포기한 이유: 외부 파일, 서비스, HTML, 이미지, 코드, 데이터와의 연동성이 떨어지고, Netior가 본질보다 저장소가 된다.

### 모든 resource를 Netior instance로 만든다

- 내용: `.md`, `.svc`, 코드 파일, 이미지, JSON 데이터, 외부 service object를 모두 Netior instance로 등록한다.
- 장점: Netior 내부에서는 모든 대상을 같은 방식으로 다룰 수 있다.
- 포기한 이유: 실제 세계의 자료 형식과 위치를 무시하게 된다. 사용자는 이미 각 resource를 다루는 도구와 workflow를 가지고 있고, Netior가 그것을 흡수할수록 비용이 커진다.

### Meaning과 field binding을 범용 도메인 엔진으로 유지한다

- 내용: 기존처럼 `meaning`, `schema_field_bindings`, DSL config가 도메인 규칙, 계산, 파생 collection, 조건부 field visibility까지 내부에서 일반화한다.
- 장점: Netior만으로 많은 것을 표현할 수 있어 보인다.
- 포기한 이유: 도메인 독립성의 압력을 Netior 내부 모델이 모두 떠안게 된다. 복잡한 field behavior는 결국 사용자가 설정하기 어렵고, 실제 domain rule은 외부 capability가 더 자연스럽게 처리할 수 있다.

### Canvas/network engine을 다시 중심에 둔다

- 내용: 모든 정의, 할당, 탐색, 관계 조작을 canvas network 위에서 시작하고 끝낸다.
- 장점: Netior라는 이름과 기존 정체성을 가장 직접적으로 이어갈 수 있다.
- 포기한 이유: instance 소유권을 낮춘 만큼, world는 directory/resource와 더 직접적으로 연결되어야 한다. Canvas는 중요한 projection이지만 저장과 탐색 전체의 엔진이 되어서는 안 된다.

## Consequences

좋아지는 점:

- Netior의 책임이 명확해진다.
- instance content 저장 부담이 줄어든다.
- resource가 원래 위치와 형식을 유지하므로 외부 도구와 연결하기 쉬워진다.
- Netior가 닫힌 저장소가 아니라 열린 ontology layer가 된다.
- 모든 파일과 데이터가 Netior-owned instance가 아니어도, 더 풍부하고 정확한 ontology를 구축할 수 있다.
- `meaning`이 단순해지고, 도메인별 규칙과 계산은 외부 capability로 보낼 수 있다.
- World/Model과 directory path의 관계가 더 중요한 설계 축으로 올라온다.
- AI는 무작정 검증자가 아니라 translator, actor, validator로 역할이 분리된다.
- 개발 로드맵을 `세계 정의 -> View -> 세계 동작/변화 -> 세계 측정 -> 세계 업데이트`로 단순화할 수 있다.

감수해야 하는 점:

- Instance와 Resource를 분리하면서 처음에는 개념 학습 비용이 있다.
- resource identity, mapping, evidence, change detection을 더 정교하게 설계해야 한다.
- Netior가 직접 소유하지 않는 resource는 외부에서 변경될 수 있으므로, 변화 측정과 해석 과정이 중요해진다.
- Meaning을 단순화하는 대신 capability/SDK/API boundary를 명확하게 만들어야 한다.
- AI와 Netior의 책임 경계를 더 엄격하게 설계해야 한다.

나중에 다시 열어볼 조건:

- 하나의 Instance가 여러 Model의 Kind를 필요로 하는 실제 사례가 많아질 때
- 파일 전체가 아니라 파일 내부 일부를 안정적으로 sub-instance로 다뤄야 하는 사례가 많아질 때
- resource mapping 없이 순수 manual value만으로도 충분한 instance가 많아질 때
- domain rule과 계산을 capability로 외부화하는 방식이 오히려 더 복잡해질 때
- directory path와 Model tree의 관계가 단일 규칙으로 설명되지 않을 때

## Implementation Notes

우선순위는 새 도메인 모델의 책임선을 단순한 기준으로 고정하는 것이다.

건드려야 할 영역:

- domain model: World, Model, Kind, Property, RelationKind, Instance, Resource
- resource mapping: resource identity, instance assignment, property value source, evidence
- domain operation: 정의, 할당, 관계, resource mapping, change detection, measurement
- meaning simplification: 기존 meaning/field binding 중 definition 계층으로 남길 것과 capability로 외부화할 것을 분리
- capability boundary: 도메인별 규칙, 계산, 검증을 외부 실행 계층으로 보낼 수 있는 계약
- migration strategy: 기존 schema/meaning/field binding 자산을 새 정의 모델로 옮길 수 있는 경로

건드리지 말아야 할 영역:

- 아직 확정되지 않은 물리 DB table/field를 과도하게 고정하지 않는다.
- Canvas를 다시 앱 전체의 유일한 중심으로 만들지 않는다.
- Property definition에 value를 넣지 않는다.
- Resource와 Instance를 같은 개념으로 합치지 않는다.
- `schema_field_bindings`식의 범용 behavior 계층을 새 meaning 모델에 그대로 재현하지 않는다.
- AI 제안을 user acceptance 없이 곧바로 truth로 저장하지 않는다.

- `docs/00-product/NETIOR_PHILOSOPHY.md`
- `docs/02-domain/NETIOR_DEFINITION_MODEL.md`
- `docs/02-domain/NETIOR_DOMAIN_OPERATIONS.md`

## Validation

이 결정은 아직 코드 검증이 아니라 설계 검증 단계다.

확인할 것:

- 여러 도메인에서 `World -> Model -> Kind/RelationKind -> Instance/Resource`가 자연스럽게 설명되는가
- `.md`, `.svc`, 코드 파일, 이미지, JSON 데이터가 Netior-owned instance가 되지 않아도 kind/relation assignment가 가능한가
- Property value가 resource에서 온 경우와 manual로 존재하는 경우를 모두 설명할 수 있는가
- Resource와 Instance를 분리했을 때 relation assignment가 더 명확해지는가
- 기존 `schema_field_bindings`가 떠안던 conditional/computed/derived behavior를 새 meaning 모델에 억지로 넣지 않아도 되는가
- 도메인별 규칙과 계산을 capability로 외부화할 때 Netior가 무엇을 요청하고 무엇을 기록해야 하는가
- AI 제안이 accepted/rejected/superseded 흐름을 가질 수 있는가

## Thinking Log

```text
초기 문제의식은 "현재 도메인 모델이 너무 복잡하고 실제 ontology를 구성하지 못한다"는 것이었다.

핵심 전환점은 instance 소유권을 내려놓는 것이었다. 이것은 Netior의 역할을 축소하는 결정이 아니라, 닫힌 저장소에서 열린 ontology layer로 바꾸는 결정이다.

Netior가 `.md`, `.svc`, 코드 파일, 이미지, 데이터 파일을 모두 자기 instance로 만들려고 하면 현실의 resource 구조를 왜곡한다. 반대로 resource는 그대로 두고, 그 resource가 어떤 Kind인지, 어떤 Instance의 근거인지, 어떤 Relation에 참여하는지를 기록하면 ontology는 더 현실에 가까워진다.

Property는 value를 가지지 않는다. Property는 메타 정의이고, value는 Instance에 속한다. 그 value는 resource에서 읽힐 수도 있고, 사용자가 직접 넣은 manual assertion일 수도 있다.

Instance와 Resource는 분리한다. Instance는 ontological target이고, Resource는 파일/폴더/URL/service object/sub-resource 같은 외부 근거다. 이 분리는 헷갈림을 만들기도 하지만, Netior를 외부 세계에 열어두기 위해 필요하다.

기존 legacy의 `schema_field_bindings`는 닫힌 Netior가 도메인 독립성을 내부에서 해결하려 했던 흔적이다. field behavior가 option source, schema composition, conditional field, computed field, derived collection까지 떠안으면서 meaning/field 체계가 과도하게 복잡해졌다.

새 방향에서는 meaning을 다시 가볍게 만든다. Meaning은 domain을 표현하는 언어이지, 모든 도메인 계산과 결정 규칙을 직접 실행하는 범용 엔진이 아니다. 도메인별 규칙과 계산은 capability/SDK/외부 실행 계층으로 열어두고, Netior는 요청, 근거, 판단, 결과 기록을 책임진다.

World의 하위 단위 이름은 Scope, Realm 등을 거쳐 Model로 결정했다. Model은 World를 Netior가 다룰 수 있도록 압축한 표현이라는 뉘앙스가 맞았다.

화면과 조작에 대한 논의도 있었지만, 이 로그의 중심은 그것이 아니다. 이번 기록에서 남겨야 할 것은 Netior가 무엇을 소유하고 무엇을 외부에 열어둘 것인지, 그리고 그 결정이 ontology의 정확성과 meaning의 단순화에 어떤 영향을 주는지다.
```
